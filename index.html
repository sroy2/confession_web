Current thoughts: best summary I've found of current capabilites

  • Ways to open databases in a web browser:
    • Indexeddb - html5
    • (depreciated) WebSQL - html5
    • (depreciated) GearsAPI - google/js
    • (currently rejected) Adobe AIR
    • silverlight (not if I'm doing it)
    • File API?

  • Non database solutions:
    • json & jquery
    • xml & xslt (or xul)
    • make the back end team send up webpages :)

With the DB solutions we can't just pass a database back and get it opened in the browser, as far as I can tell (need to explore FileAPI to be sure), we need to build the DB locally. This can be done using the NonDB solutions. But, if we are going to use a NonDB method to transport the data back is it worth importing it into a DB just for the convenience of being able to run sql queries? We could also directly load the information from the flat files sent back. The DB is nice because it allows us to store information it a slightly more logical way (the DBs can be persistant between browser sessions) and avoids a lot of naming scheme issues that would undoubtably come by having a lot of flat files laying around, but the DB can be limited by size. Most browsers limit Local Storage to 5MB (although some like FF will prompt to allow the user to use more space).

Currently I am going to proceed under the assumption that we will at a minimum be using json to get he data back to the front end. From there we can modify the design to either use load into a local DB (if we want to store the information for comparisons or just use sql queries) or just load directly and either come up with a naming convention or not worry about saving results.
